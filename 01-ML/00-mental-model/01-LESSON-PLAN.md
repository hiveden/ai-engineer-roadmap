## 一、为什么学（时代语境 · 3 条核心论点）

### 1.1 传统 ML 是 LLM 时代的基础设施

把任何一个"高级"的 LLM/Agent 应用拆开，里面全是传统 ML 的零件：

```
Agent 应用
├── LLM（大脑）              ← Transformer，本质是神经网络
├── Embedding（向量化）       ← 一个学过的函数 word → vector
├── Vector Search（检索）     ← 本质就是 KNN
├── Reranker（重排）          ← 一个分类/打分模型
├── 意图识别（路由）           ← 小型分类模型（决策树 / 逻辑回归足够）
├── 内容审核                   ← 分类模型
└── 业务监控                   ← 时序异常检测（聚类 / 统计）
```

**一个 RAG 用 5 行 LangChain 代码调用，底下藏着 7 个 ML 模型在协作**。不懂 ML，出问题连 debug 入口都找不到。

### 1.2 生产系统里传统 ML 仍是主力

| 业务场景 | 主流方案 | 为什么不用 LLM |
|---|---|---|
| 银行信用卡反欺诈 | XGBoost / LightGBM | 延迟毫秒 vs 秒、不可解释、监管不接受 |
| 电商商品推荐排序 | XGBoost + CTR 模型 | 成本：10 亿次/天，LLM 烧不起 |
| 信贷评分卡 | 逻辑回归 + 决策树 | 监管要求"每个决策必须能解释" |
| 实时风控 | 决策树 + 规则引擎 | 亚毫秒延迟，LLM 物理做不到 |
| 用户分群 / 冷启动 | K-Means + 协同过滤 | 几十行代码搞定 |
| 时序异常检测 | ARIMA / Prophet / XGBoost | 解释性 + 成本 |

**量化对比**：LLM 做一次欺诈判断 ≈ 200ms + $0.001；XGBoost 做一次 ≈ 1ms + $0.0000001 —— **200 倍延迟差 + 10000 倍成本差**。

### 1.3 跨语言部署是后端工程师的护城河

稀缺能力：
> "把 sklearn 训出的模型，导出成 ONNX，在 Java Spring Boot 里加载，跑出 1ms 的 P99 延迟，扛住 10 万 QPS，还能实时监控模型衰退。"

所需能力栈：
- ✅ Python ML 训练
- ✅ 模型工件序列化（ONNX / PMML）
- ✅ Java / Go 后端工程能力（**后端工程师的存量**）
- ✅ 高并发系统设计（**存量**）
- ✅ 监控 / 灰度 / 回滚（**存量**）

后三项是后端开发经验的存量。**传统 ML 是这个杠杆的支点**。

### 信念校准（学习焦虑发作时回来读）

1. LLM 没消灭传统 ML，它把传统 ML 变成了基础设施
2. 基础设施层永远需要懂底层的人
3. 真正的 Agent 系统是"LLM + 传统 ML + 规则引擎"的混合体
4. 护城河不是"会调 LLM"，而是"懂 ML + 会工程化落地"
5. 传统 ML 是建立"模型直觉"的最快路径，没有捷径

## 二、怎么学（方法论）

### 2.1 四层学习法（横向阶段）

```
第 1 层 · 认知层（1 周）    搞清 10 个核心概念，不写代码
    ↓
第 2 层 · 接口层（2 周）    学 sklearn API，三步 fit/predict
    ↓
第 3 层 · 场景层（边学边累） 背"什么算法配什么业务场景"
    ↓
第 4 层 · 落地层（主场）     Python 训练 → ONNX/PMML → Java/Go 加载
    ↓
⚠️ 数学层放最后             等做调优 / 读论文时再补
```

sklearn 所有算法都是**同一个三步模式**：

```python
model = SomeAlgorithm(超参数)   # 1. 实例化
model.fit(X_train, y_train)      # 2. 训练
y_pred = model.predict(X_test)   # 3. 推理
```

**换算法 = 换第 1 行**，其他不动。

### 2.2 L1-L4 深度等级（纵向投入）

4 层是"现在该学哪一层"，每层内部还要决定"这个主题投入多深"。

| 等级 | 动作 | 单项时间感 | 适用场景 |
|---|---|---|---|
| **L1 扫概念** | 读章节，知道是什么、解决什么问题 | 15-30 min | 离方向远 / 有更好工具替代 / 知道存在即可 |
| **L2 跑现成** | 跟着跑 demo，看输入输出 | 30-60 min | 需要手感但不需要精通 |
| **L3 拆步骤** | 拆 step 脚本 + 吃透注释 + 能讲流程 | 1-2 h | 和工作日常相关 / 某类任务原型 |
| **L4 精做+变体** | L3 基础上换数据、改参数、能讲给别人听 | 半天-一天 | 核心战场 / 反复用 |

**使用原则**：
- 起始校准，不是死规定；做到一半可升级或降级
- 打不出等级 = 不够了解业务价值 → 先读章节概述再打
- **第一次课的目标通常是 L2，L3 通过多轮复习达成**

### 2.3 阶段一算法的深度配置示例

> **初始校准**，做到每一项时凭直觉调整。

| 主题 | 倾向等级 | 理由 |
|---|---|---|
| 线性回归 | L3 | 回归任务原型 |
| 逻辑回归 | L3 | 分类任务原型 |
| KNN | L2 → L3 | 落地用 Vector DB 时升级 |
| 决策树 | L3 | 可解释性新维度 |
| 随机森林 | L2 | 决策树延伸 |
| XGBoost | L3 | 表格数据工业界主力 |
| K-Means | L3 | 无监督思路切换 |
| 朴素贝叶斯 | L1 | 概念框架重要，落地场景少 |
| PCA 降维 | L2 | Embedding 可视化会用 |
| 时序（电力负荷） | L1 | 离核心方向远 |

## 三、学什么（A/B/C 组 · 考试范围）

下面三组是"考试范围"，其他查文档即可。

### 3.1 A 组 · 10 个概念（附 RDBMS 映射）

| 概念 | 一句话 | RDBMS 对应（工程类比） |
|---|---|---|
| **特征 Feature** | 模型的输入字段 | 业务字段 Column（`user_age`、`click_count`） |
| **标签 Label / Target** | 模型要预测的目标 | 待预测的目标字段（`is_fraud`） |
| **样本 Sample** | 一条数据 | 表里的一行记录 Row |
| **数据集 Dataset** | 所有样本的集合 | 数据库表 Table（通常是宽表） |
| **训练集 / 验证集 / 测试集** | 训练 / 调参 / 最终评估 | 数据切 3 份，**不能混** |
| **监督 vs 无监督** | 有 label vs 没 label | 聚类是无监督 |
| **分类 vs 回归** | 输出离散类别 vs 连续数值 | `Enum/Boolean` vs `Float` |
| **过拟合 Overfitting** | 模型死记训练集，上线翻车 | HashMap 死记硬背，没学规律 |
| **欠拟合 Underfitting** | 模型太蠢，连训练集都学不会 | 用一条直线拟合一个圆 |
| **偏差 / 方差权衡** | 太简单 vs 太复杂的取舍 | — |
| **超参数 vs 参数** | 手动设的（`max_depth=8`） vs 模型自己学的 | — |
| **交叉验证 Cross-Validation** | 切 K 份轮流验证，避免运气 | — |

### 3.2 B 组 · 5 个工程坑（生产事故防坑指南）· 📎 未实操 TODO

> **5 项全都没实操踩过**——认识名字 + 能讲原理即可。等做真实生产项目时回来逐个打勾。

- [ ] **状态漂移**：特征预处理的 mean / std / encoder 映射表**必须随模型一起序列化**，否则线上推理和训练对不上
- [ ] **训推一致性**：训练时和线上推理的特征 schema **字段名、类型、顺序必须完全一致**——很多事故是 Python 训练和 Java 推理时字段顺序不同
- [ ] **类别不平衡**：99% 通过 + 1% 欺诈，模型偷懒全猜"通过"，准确率 99% 但没用——必须 SMOTE 或加权处理
- [ ] **特征泄露 Data Leakage**：训练时用了"未来才能知道的字段"，离线完美上线翻车
- [ ] **模型衰退 Model Drift**：上线后数据分布会变，必须**持续监控 + 定期重训**

### 3.3 C 组 · 算法 ↔ 业务场景表（背下来）

| 算法 | 黄金场景 | 千万别用 |
|---|---|---|
| **线性回归** | 数值预测 baseline | 复杂非线性关系 |
| **逻辑回归** | 二分类、风控评分卡 | 复杂决策边界 |
| **决策树** | 可解释性强（金融审计） | 高维稀疏数据 |
| **随机森林** | 表格数据、要稳定 | 文本 / 图像 |
| **XGBoost / LightGBM** | **表格数据 SOTA、Kaggle 王者** | 文本 / 图像 |
| **KNN** | 向量检索、推荐冷启动 | 高维数据（维度灾难） |
| **K-Means** | 用户分群、冷启动 | 非凸聚类 |
| **朴素贝叶斯** | 文本分类（垃圾邮件） | 特征强相关 |
| **神经网络 / 深度学习** | 文本、图像、语音 | 表格数据（打不过 XGBoost） |
| **LLM / Transformer** | NLU、生成、Agent | 简单分类（杀鸡牛刀） |

背下这张表 → 90% 的需求评审会都能给出正确选型判断。

## 四、怎么用（工程落地架构）

### 4.1 离线训练 / 在线推理 · 双链路

```
【离线侧 · Python 主场】
  历史数据（MySQL/Hive/Kafka）
       ↓
  特征工程（清洗、标准化、独热编码）
       ↓
  训练（sklearn / PyTorch）
       ↓
  模型文件（.onnx / .pmml）
       ↓
  推送到模型仓库（OSS/S3）

【在线侧 · Java/Go/Node 主场】
  HTTP/gRPC 请求
       ↓
  实时聚合特征（Redis 缓存 + MySQL 历史）
       ↓
  加载模型 → model.predict()
       ↓
  返回结果 + 监控埋点
```

### 4.2 三种部署架构（工业界选型）

| 架构 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **Python Wrapper** | 模型包在 FastAPI/Flask 里，业务服务 HTTP/gRPC 调 | 简单、调试方便 | 多一层网络开销、Python 并发弱 |
| **ONNX Runtime** 👑 | 模型导出 ONNX，Java/Go/Node 引入 `onnxruntime-*` **直接内存加载** | 毫秒级、零网络延迟、复用现有高并发架构 | 需掌握 ONNX 导出 + 各语言 SDK |
| **前端 / 边缘计算** | TensorFlow.js / WebAssembly，模型发给浏览器 | 减轻后端压力、实时响应（人脸活体检测、视频滤镜） | 模型需足够小 |

**工业界主流 = ONNX Runtime**（跨语言原生加载）。这是后端工程师的护城河战场。

### 4.3 特征工程五分支

**特征工程 = 把业务数据转成模型能吃的数字向量**——算法工程师 80% 的时间在做这个。

| 分支 | 传统定义 | 开发者视角 |
|---|---|---|
| **特征提取 Feature Extraction** | 非结构化数据转特征向量 | **反序列化 + 编码**：图片 → `[224,224,3]` RGB 数组；文本 → tokenize 成 `[1,512]` 整数数组 |
| **特征预处理 Preprocessing** | 归一化、标准化，消除量纲影响 | **数据对齐**：月薪 3k-100k vs 年龄 18-60 → 都压到 `[0,1]`，防止量纲碾压 |
|  &nbsp;&nbsp;&nbsp;&nbsp;├ 归一化 Normalization (Min-Max) | `(x-min)/(max-min)` | 缩到 `[0,1]` |
|  &nbsp;&nbsp;&nbsp;&nbsp;└ 标准化 Standardization (Z-score) | `(x-mean)/std` | 均值 0、标准差 1 |
| **特征降维 Dimensionality Reduction** | PCA 主成分分析 | **删冗余列优化性能**：1 万列 → 100 列（解决维度灾难） |
| **特征选择 Feature Selection** | 保留最有用的 K 个 | 类似建 MySQL 索引前分析选择度 |
| **特征组合 Feature Cross** | 多个特征交叉 | **拼接联合字段**：`品类=美妆` × `性别=男` → `美妆_男`（推荐系统标配） |

### 4.4 存量 vs 新学

**已会的直接复用**：MySQL / Redis（特征存储）、Kafka（实时特征流）、gRPC/SSE（推理接口）、Docker / k8s（部署）、Prometheus（监控）。

**需要新学的只有 4 件事**：
1. 训练侧 API（sklearn / PyTorch / HuggingFace）
2. 模型工件格式（ONNX / PMML）
3. 推理侧 SDK（`onnxruntime-java` / `onnxruntime-node`）
4. **训推一致性**——线下训练和线上推理的特征处理逻辑必须完全一致，这是大坑
