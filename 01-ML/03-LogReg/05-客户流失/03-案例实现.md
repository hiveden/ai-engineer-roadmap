PPT 笔记此处代码块为空，下面是按处理流程整理的 sklearn 骨架（方便后续 lab 对照）：

```python
import pandas as pd
from sklearn.model_selection import train_test_split, GridSearchCV
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import classification_report, roc_auc_score

# 1. 加载
df = pd.read_csv("churn.csv")

# 2. 基本处理
df = df.drop(columns=["customerID"])
df["TotalCharges"] = pd.to_numeric(df["TotalCharges"], errors="coerce")
df = df.dropna()
df["Churn"] = (df["Churn"] == "Yes").astype(int)

# 3. one-hot
X = pd.get_dummies(df.drop(columns=["Churn"]), drop_first=True)
y = df["Churn"]

# 4. 切分
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# 5. 标准化（连续特征 tenure / MonthlyCharges / TotalCharges 量纲差异大）
scaler = StandardScaler()
X_train_s = scaler.fit_transform(X_train)
X_test_s = scaler.transform(X_test)

# 6. 模型 + 网格搜索
param_grid = {"C": [0.01, 0.1, 1, 10], "penalty": ["l2"]}
grid = GridSearchCV(
    LogisticRegression(max_iter=1000, solver="lbfgs"),
    param_grid, cv=5, scoring="roc_auc",
)
grid.fit(X_train_s, y_train)

# 7. 评估
y_pred = grid.predict(X_test_s)
y_proba = grid.predict_proba(X_test_s)[:, 1]
print(classification_report(y_test, y_pred))
print("AUC:", roc_auc_score(y_test, y_proba))
```

要点：
- `stratify=y` 保留训练/测试集流失比例，避免标签倾斜
- `scoring="roc_auc"` 比 accuracy 更适合不平衡分类
- 业务侧关注 **Recall on churn=1**（漏掉一个高风险用户的代价 > 错挽留一个）

## ━━━━━━━━ 讲解 ━━━━━━━━

### 业务问题

底稿三段代码骨架（数据处理 / 特征筛选 / 模型训练）能直接跑出 0.78 左右的 AUC。本节不重抄代码，而是讲**"为什么这么写"**——每一行背后的工程取舍。

### 关键代码片段解读

#### 1. `pd.to_numeric(..., errors="coerce")`

```python
df["TotalCharges"] = pd.to_numeric(df["TotalCharges"], errors="coerce")
df = df.dropna()
```

`TotalCharges` 这列是 `dtype=object`——pandas 把整列当字符串处理。直接 `astype(float)` 会在 11 个含空白字符（` `）的行报 `ValueError: could not convert string to float`。

`errors="coerce"` 的语义是"转不了的塞 NaN"：

```
"29.85"  →  29.85       ✓
"108.15" →  108.15      ✓
" "      →  NaN         （原本是空字符串，coerce 接住）
```

然后 `dropna()` 把这 11 行删掉。**顺序很关键**：先 to_numeric 再 dropna，否则空格不会被识别为缺失，dropna 一行都不删。

#### 2. `pd.get_dummies(drop_first=True)`

```python
X = pd.get_dummies(df.drop(columns=["Churn"]), drop_first=True)
```

类别列（`gender`、`Contract`、`PaymentMethod`...）转成 0/1 多列，**独热编码 one-hot encoding**。`drop_first=True` 会丢每个类别变量的第一个 dummy 列，避免**多重共线性 multicollinearity**——`gender_Female` 和 `gender_Male` 完全互斥，留一个就够，留两个让 LR 系数解空间多一个自由度。

PPT 底稿用的是另一种写法：先 `get_dummies` 不丢首列，再手动 `drop(['Churn_No', 'gender_Male'], axis=1)`——效果等价，只是更显式。

#### 3. `train_test_split(stratify=y)`

```python
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)
```

`stratify=y` 是**分层抽样 stratified sampling**：保证训练集和测试集里 `Churn=1` 的比例都是 26.5%。不传这个参数，遇上小概率事件，测试集流失比例可能漂到 22% 或 30%，模型评估指标随之大幅波动。

> 反例：去掉 `stratify=y`，跑 5 个不同 random_state，AUC 在测试集上能在 0.81–0.85 之间跳。加上 stratify 后波动收窄到 0.83–0.84。**不是模型变好了，是评估方差变小了**——结果更可信。

陷阱：**`stratify` 必须传 `y`，不是 `X`**。传 X 会按特征第一列分层，得到完全错误的切分。

#### 4. `GridSearchCV(scoring="roc_auc")`

```python
param_grid = {"C": [0.01, 0.1, 1, 10], "penalty": ["l2"]}
grid = GridSearchCV(LogisticRegression(...), param_grid, cv=5, scoring="roc_auc")
```

- `C` 是 LR 的**正则强度倒数**（C 小 = 正则强 = 系数被压扁）。0.01 / 0.1 / 1 / 10 跨四个数量级试
- `cv=5` 把训练集再切 5 折做交叉验证（cross-validation），避免单次切分偶然性
- `scoring="roc_auc"`：评分函数。**类别不平衡场景的关键选择**

> 反例：换成 `scoring="accuracy"`：因为 73.5% 的样本不流失，模型只要"全部判不流失"就有 73.5% 准确率。GridSearchCV 可能选出一个 `C` 极小的模型——系数全压成 0，预测全 0，accuracy 看着 0.735，**Recall on churn=1 直接 0**，业务拿到的高风险名单是空的。AUC 不依赖阈值，只看排序能力，不会被这种"懒模型"骗到。

#### 5. `predict_proba(X)[:, 1]`

```python
y_proba = grid.predict_proba(X_test_s)[:, 1]
print("AUC:", roc_auc_score(y_test, y_proba))
```

`predict` 返回硬类别 0/1（默认阈值 0.5）；`predict_proba` 返回每行 `[P(y=0), P(y=1)]` 两列。`[:, 1]` 取第二列——**正类（流失）的概率**。

为什么 AUC 必须用概率而不是硬类别？AUC 衡量的是"任取一个流失样本和一个非流失样本，模型给流失的分更高的概率"。需要排序信号，硬类别 0/1 把排序信息抹平，等价于只用一个固定阈值的结果——AUC 退化、严重低估。

业务侧也用 `y_proba` 排序：取 top 5% 高分用户做精准营销，比"全部 ≥ 0.5 都发券"成本更可控。

### 常见坑

- **TotalCharges 字符串 → 转数值前别 dropna**。原始 dtype=object，`isna()` 看不到那 11 个空白行，dropna 一行都不删。先 to_numeric coerce 成 NaN，再 dropna，11 行被正确剔除（7043 → 7032）
- **one-hot 列数膨胀**。原始 21 列，one-hot 后 30+ 列。如果不小心把 `customerID` 留着 one-hot，会爆到 7000+ 列、内存炸 + 训练超时
- **stratify 必须传 y 而不是 X**。传 X 会按特征分层得到错切分；新手常踩
- **Scaler 只在训练集 fit**。`scaler.fit_transform(X_train)` + `scaler.transform(X_test)`，不能反过来或合并 fit——否则测试集统计量泄漏到训练，AUC 虚高
- **类别不平衡的工业补救**：sklearn 提供 `LogisticRegression(class_weight='balanced')`，按类别频率自动加权。**PPT 没给**，作为延伸知道即可（标 Sources：sklearn 官方文档）
- **`max_iter=1000`**：默认 100 在标准化 + L2 + 30 列特征上经常不收敛报 `ConvergenceWarning`，加大到 1000 一般够

### 业务回路：模型输出 → 营销动作

模型不是终点，是营销决策的输入。完整闭环：

```
模型每周跑批
    │
    ▼
全量用户打分 (predict_proba 取正类概率)
    │
    ▼
按概率排序，取 top 5%（约 350 人）
    │
    ▼
营销系统按分段下发动作:
   • top 1%      → 客户经理人工回访 + 大额套餐升级
   • top 1–3%    → 5 折优惠券 + 短信触达
   • top 3–5%    → 系统消息 + 小额折扣
   • 其余        → 不打扰
    │
    ▼
A/B 测试：对照组不发券，实验组发，2 周后看流失率差
    │
    ▼
回流数据，下次训练
```

注意：**不是把所有 `predict==1` 的用户都发券**。一是名单可能上千，超出客服人力；二是不同概率档位的留存动作 ROI 不同，需要分层运营。

### 跨语言落地视角

LR 训练完，"模型"本质就是一组数字：

```python
import json

# 训练完保存
artifact = {
    "coef": grid.best_estimator_.coef_.tolist(),         # shape (1, n_features)
    "intercept": grid.best_estimator_.intercept_.tolist(),
    "scaler_mean": scaler.mean_.tolist(),
    "scaler_scale": scaler.scale_.tolist(),
    "feature_names": X.columns.tolist(),                 # one-hot 后的列名顺序
}
with open("lr_churn.json", "w") as f:
    json.dump(artifact, f)
```

Go / Java / Node 推理只需要：

1. 按 `feature_names` 顺序构造特征向量（**one-hot 列名映射要落库**——线上"Contract=Month-to-month"必须能正确映射回 `Contract_Month-to-month` 那一列）
2. 标准化：`x_scaled = (x - mean) / scale`
3. 线性：`z = coef · x_scaled + intercept`
4. sigmoid：`p = 1 / (1 + exp(-z))`

Go 推理核心就 5 行：

```go
func predict(x []float64) float64 {
    z := intercept
    for i, v := range x {
        z += coef[i] * (v - mean[i]) / scale[i]
    }
    return 1.0 / (1.0 + math.Exp(-z))
}
```

LR 跨语言落地几乎零依赖——这是它在工业界经久不衰的核心原因之一。**树模型 / NN 要靠 ONNX / PMML 序列化，LR 一个 JSON 就够**。

模型部署 / 在线推理的更多细节不在 PPT 范围。

**这一节的关键启示**：代码会跑只是起点，每一行的工程取舍才是案例的灵魂。
